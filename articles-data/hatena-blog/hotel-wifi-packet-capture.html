「ホテルのフリーWi-Fiは危険」とよく言われるが、具体的に何がどう危険なのかを技術的に説明しているサイトは少ない。

筆者はIT業界に10年以上身を置き、エンジニアとしても5年以上の経験がある。「危険だ」と言うからには、実際にパケットキャプチャで何が見えるのかを検証すべきだと考え、自宅に検証環境を構築してテストしてみた。

結論から言うと、想像以上に情報が見える。

検証環境と手法

本記事の検証は、すべて筆者が自宅に構築したテスト環境で行っている。実際のホテルWi-Fiを盗聴したわけではない。

構築した環境

  • ダミーアクセスポイント（AP）： Raspberry Piにhostapdを導入し、暗号化なし（Open）のWi-Fiネットワークを構築
  • パケットキャプチャツール： Wireshark（業界標準のネットワーク解析ツール）
  • テスト端末： iPhone（iOS最新版）、Android端末（Pixel）
  • テストしたアプリ： ブラウザ（Safari/Chrome）、メールアプリ、チャットアプリ、地図アプリ

この環境は「ホテルが提供するパスワードなしのフリーWi-Fi」と技術的に同一の条件だ。同じネットワーク上にいる第三者（=筆者のPC）から、接続端末の通信がどこまで見えるかを検証した。

暗号化なしのWi-Fiで何が見えるのか

見えたもの①：DNS クエリ（アクセス先のドメイン名）

スマホでブラウザを開くたびに、DNSクエリがキャプチャされる。これはどのサイトにアクセスしようとしているかがわかるということだ。

例えば、テスト端末でGoogleを開くと以下のようなDNSクエリが平文で流れる。

Query: www.google.com  Type: A
Query: mail.google.com  Type: A
Query: maps.google.com  Type: A

HTTPSで暗号化されていても、DNS クエリは暗号化されない（DNS over HTTPSやDNS over TLSを明示的に設定していない限り）。つまり、同じWi-Fiにいる攻撃者は「この人がどのサイトを見ているか」のリストを取得できる。

見えたもの②：接続先のIPアドレスとポート番号

DNSだけでなく、TCP接続のSYNパケットから接続先のIPアドレスとポート番号も見える。IPアドレスを逆引きすれば、使っているサービスを特定できる。

見えたもの③：HTTP通信の中身（暗号化なしの場合）

2026年現在、ほとんどのサイトがHTTPSに対応しているため、通信の中身（ログイン情報やメール本文）が平文で流れるケースは少ない。

しかし、一部のIoT機器のAPI通信や、古い業務システムのWebインターフェースはまだHTTPを使っている場合がある。出張先でこういったシステムにアクセスする人は要注意だ。

見えなかったもの

HTTPS通信の中身（リクエストボディ、Cookieの値、ログイン情報等）は暗号化されており、見えなかった。これはTLS/SSLが正常に機能している証拠だ。

つまり現代のインターネットでは「通信の中身」は守られているが、「どこと通信しているか」は丸見えというのが実態だ。

HTTPS通信でも漏れる情報がある

「HTTPSなら安全」と思っている人が多いが、それは半分しか正しくない。HTTPSで暗号化されても以下の情報はキャプチャ可能だ。

情報 HTTPS未使用 HTTPS使用
通信の中身（パスワード等）見える見えない
アクセス先ドメイン名（DNS）見える見える
接続先IPアドレス見える見える
TLSハンドシェイクのSNI—見える
通信のデータ量・タイミング見える見える

SNI（Server Name Indication）は特に注意すべきだ。HTTPS接続の初期段階（TLSハンドシェイク）で、接続先のドメイン名が平文で送信される。ECH（Encrypted Client Hello）が普及すればこの問題は解消されるが、2026年時点ではまだ広く使われていない。

つまり、ホテルのフリーWi-Fiで作業すると「この人は今、○○銀行のサイトにアクセスした」「転職サイトを見ている」「医療機関のサイトにアクセスした」程度の情報は、同じネットワーク上の第三者に筒抜けになる可能性がある。

VPNを通すと何が変わるか？Before / After

同じテスト環境で、VPN接続時のキャプチャ結果と比較した。使用したのはNordVPN（NordLynxプロトコル）。

情報 VPNなし VPNあり
DNSクエリ平文で見える見えない（VPNトンネル内）
接続先IPアドレス実際のサイトIPVPNサーバーのIPのみ
SNI見える見えない（トンネル内）
通信の中身HTTPSで暗号化二重に暗号化
第三者から見える情報アクセス先のリスト「VPNを使っている」ことだけ

VPNを通すと、第三者から見えるのは「VPNサーバーとの間で暗号化された通信が流れている」という事実だけだ。どのサイトにアクセスしたか、どんなデータを送受信したかは一切わからない。

これが「フリーWi-FiではVPNを使え」と言われる技術的な根拠だ。なんとなく「安全になる」のではなく、メタデータ（通信先情報）を含むすべての通信がVPNトンネル内に隠蔽される。

海外ホテルWi-Fiを安全に使うための3つの設定

パケットキャプチャの検証結果を踏まえ、エンジニアとして推奨する設定は以下の3つだ。

1. VPNを常時ONにする

NordVPN・Surfshark・ExpressVPNなどの主要サービスには「自動接続」機能がある。Wi-Fiに繋いだ瞬間に自動でVPNが起動する設定にしておく。

設定方法：アプリの設定 → 自動接続 → 「Wi-Fiネットワーク接続時」をONにする。

2. DNS over HTTPS（DoH）を有効にする

VPNを使えない状況でも、DNSクエリだけは暗号化しておきたい。

  • iPhone： 設定 → Wi-Fi → 接続中のネットワーク → DNSを手動設定 → 1.1.1.1（Cloudflare）
  • Android： 設定 → ネットワーク → プライベートDNS → 「dns.google」または「1dot1dot1dot1.cloudflare-dns.com」

3. 自動Wi-Fi接続をOFFにする

スマホが過去に接続したWi-Fiに自動で繋ぐ機能は便利だが、同名のSSIDを持つ偽APに接続してしまうリスクがある（Evil Twin攻撃）。

特に海外では「Hotel_WiFi」「Free_WiFi」のような汎用的なSSIDが多い。自動接続は切り、都度手動で接続する方が安全だ。

ホテルWi-Fiのリスクと具体的な対策手順はこちらの記事にさらに詳しくまとめた。今回の検証は自宅環境だが、実際のホテルWi-Fiで起き得るリスクとその防ぎ方を網羅的に解説している。

まとめ

今回の検証で明らかになったのは以下のことだ。

  • 暗号化なしのフリーWi-Fiでは、どのサイトにアクセスしたかが第三者に見える
  • HTTPSが有効でも、DNS クエリとSNIは平文で漏れる
  • VPNを通すと、メタデータを含むすべての通信が隠蔽される
  • 「なんとなく危険」ではなく、パケットキャプチャで実際に確認できるレベルの問題

「ホテルWi-Fiは危険」は漠然とした警告ではない。技術的に検証可能な、明確なリスクだ。特に海外のホテルでは、ネットワーク管理者がどの程度のセキュリティ対策をしているか不透明なケースが多い。

VPNをONにする。たった一手間で、リスクは劇的に下がる。
